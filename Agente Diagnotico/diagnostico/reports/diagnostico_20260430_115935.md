## 1. NÍVEL DE COMPLEXIDADE

**Classificação:** 🚨 Bomba

**Justificativa:** A complexidade é classificada como "Bomba" devido a dois fatores críticos identificados e a um vasto leque de incertezas. Primeiramente, a extração de dados de "cerca de 50 layouts diferentes" de PDFs de planos de saúde é um desafio técnico com altíssimo risco de estouro de prazo, custo e acurácia. A experiência passada com PDFs despadronizados que quebraram em produção serve como alerta direto. Cada layout pode exigir regras ou modelos de extração específicos, tornando a manutenção e a criação inicial extremamente trabalhosas. O risco de PDFs malformados, ilegíveis ou com novos layouts não previstos é constante e alto.

Em segundo lugar, a lógica de "filtrar os planos para que retornasse os 3 que se encaixariam melhor no perfil" é um requisito de negócio que pode ser significativamente mais complexo do que o cliente imagina, exigindo regras de negócio sofisticadas, ponderação de critérios, e potencialmente abordagens de Machine Learning, que introduzem seus próprios riscos de acurácia, treinamento e validação.

Por fim, a maioria das áreas críticas de risco para um projeto desta natureza (integrações, autenticação, volume, LGPD, prazos, operação pós-entrega) sequer foram investigadas. Esta falta de visibilidade, somada aos desafios conhecidos, cria um cenário de projeto com altíssima probabilidade de insucesso ou de sangrar orçamentariamente.

## 2. STACK E ARQUITETURA RECOMENDADAS

### 2.1 Tecnologias Específicas
*   **Extração de PDFs (Backend):** Python é a linguagem mais indicada devido ao seu vasto ecossistema para processamento de dados, Machine Learning e OCR (ex: `pdfminer.six`, `PyPDF2`, `Tesseract-OCR` para PDFs escaneados). Para gerenciar os múltiplos layouts, é recomendável avaliar frameworks de extração de documentos como `DocParser`, ou considerar a integração com serviços de Document AI (ex: Google Cloud Document AI, AWS Textract) que podem lidar com a variabilidade de layouts de forma mais robusta, embora com custo adicional. Uma alternativa é a construção de modelos de Machine Learning personalizados para detecção de layout e extração de campos-chave, o que demandaria tempo e dados para treinamento.
*   **Backend (API/Serviços):** Python com um framework robusto para APIs como `FastAPI` (para performance e validação de dados) ou `Django REST Framework` (para um ecossistema mais completo, se houver necessidade de admin). A escolha deve priorizar a integração com as bibliotecas de processamento de PDFs e a capacidade de escalar para o processamento assíncrono.
*   **Banco de Dados:** `PostgreSQL` é a recomendação principal. Oferece robustez para dados estruturados dos planos de saúde e suporta JSONB para flexibilidade, caso os dados extraídos dos PDFs inicialmente tenham um esquema variável. Sua capacidade de indexação e busca complexa é fundamental para a funcionalidade de filtragem.
*   **Filas de Mensagens:** `RabbitMQ` ou `Apache Kafka` para orquestrar o processamento assíncrono de PDFs. Isso garante que a extração, que pode ser demorada e sujeita a falhas, não bloqueie a API principal e permite retentativas e processamento em background.
*   **Frontend:** `React` ou `Vue.js` para uma aplicação web dinâmica e responsiva, que permita aos usuários fazer upload de PDFs e interagir com os filtros de forma eficiente.

### 2.2 Arquitetura de Componentes
A arquitetura proposta é baseada em microsserviços ou um monolito bem-modularizado, com foco na robustez do fluxo de dados:
1.  **Serviço de Ingestão/Upload:** Responsável por receber os arquivos PDF (via API REST do frontend). Valida o arquivo e o armazena temporariamente, disparando uma mensagem para a fila de processamento.
2.  **Fila de Processamento de PDFs:** Uma fila de mensagens (RabbitMQ/Kafka) para desacoplar o upload do processamento intensivo.
3.  **Serviços de Extração de Dados (Workers):** Múltiplos workers (ou um serviço escalável) que consomem mensagens da fila. Cada worker é responsável por:
    *   **Classificação de Layout:** Identificar qual dos 50+ layouts o PDF corresponde. Pode ser via regras heurísticas, OCR/visão computacional ou modelo de ML.
    *   **Extração de Dados:** Aplicar a lógica específica para o layout identificado, extraindo os campos relevantes.
    *   **Normalização/Validação:** Converter os dados extraídos para um formato padronizado e realizar validações básicas.
    *   **Persistência:** Salvar os dados extraídos no banco de dados.
    *   **Registro de Erros:** Reportar PDFs que falharam na extração para revisão manual.
4.  **Serviço de Gerenciamento de Dados/Planos:** API REST que expõe os dados dos planos de saúde, permitindo operações CRUD e, mais importante, a funcionalidade de filtragem.
5.  **Serviço de Regras de Negócio/Filtragem:** Contém a lógica de negócios para o "encaixe" dos planos, recebendo parâmetros do usuário e retornando os 3 melhores. Pode ser um componente do serviço de gerenciamento ou um microsserviço separado.
6.  **Banco de Dados:** Armazena os planos de saúde extraídos e padronizados, além de informações de usuário e metadados de processamento.
7.  **Aplicação Frontend:** Interface web para upload, consulta, filtragem e visualização dos planos.

### 2.3 Possíveis Armadilhas de Arquitetura
*   **Subestimar a variabilidade dos PDFs:** Assumir que 50 layouts são fixos ou que um único extrator genérico funcionará bem. A realidade é que haverá variações dentro dos layouts e novos layouts surgirão.
*   **Falta de estratégia para dados ruins:** Não ter um mecanismo claro para lidar com PDFs que falham na extração, resultando em dados incompletos ou incorretos no sistema.
*   **Lógica de filtragem engessada:** Implementar a lógica de "melhores 3" de forma rígida, dificultando a manutenção ou a evolução dos critérios de negócio.
*   **Escalabilidade do processamento:** Não prever que o processamento de PDFs pode ser um gargalo de performance, resultando em filas longas e atrasos para o usuário.
*   **Ausência de feedback humano no loop:** Sem um mecanismo para revisão e correção dos dados extraídos, a qualidade dos dados pode degradar rapidamente.

### 2.4 Alternativas Consideradas e Descartadas
*   **Linguagem de Backend Java/C#:** Descartada devido ao maior overhead de desenvolvimento para o tipo de projeto (data processing, ML/AI), onde Python se destaca pelo ecossistema e agilidade para prototipagem e implementação de rotinas de extração complexas.
*   **Banco de Dados NoSQL (MongoDB/DynamoDB):** Descartado como primário para os dados dos planos devido à necessidade de consultas complexas e filtragem com lógica de negócios que se beneficiam de esquemas relacionais bem definidos e otimizações de índices do PostgreSQL. Pode ser considerado para armazenamento de logs ou dados temporários, mas não para o core dos planos.
*   **Extração manual ou terceirizada:** Embora possa ser uma alternativa para mitigar riscos técnicos, não atende ao requisito de automação do projeto e introduz custos operacionais contínuos não escaláveis.

## 3. PRINCIPAIS RISCOS E PONTOS DE ATENÇÃO

1.  **Extração de PDFs despadronizados (50+ layouts):**
    *   **O que é:** O maior risco. Cada um dos 50+ layouts pode ter variações e exigir regras de extração distintas. A criação e manutenção desses "parsers" é custosa e demorada.
    *   **Por que é risco:** Alta probabilidade de erros na extração (dados incorretos, incompletos), dificuldade em lidar com novos layouts ou alterações nos existentes, alto custo de desenvolvimento e manutenção, impactando a acurácia dos dados e a credibilidade do sistema. Este ponto é o principal histórico de "projetos bomba".
    *   **Impacto:** Dados inválidos no sistema, retrabalho massivo, estouro de prazo e orçamento, insatisfação do cliente, necessidade de validação humana constante.

2.  **Definição e Evolução da Lógica de Filtragem ("Melhores 3 Planos"):**
    *   **O que é:** A lógica para determinar o "melhor encaixe" é subjetiva e complexa, podendo evoluir com o tempo e os critérios de negócio do cliente.
    *   **Por que é risco:** Se não for bem definida e validada exaustivamente, pode levar a resultados insatisfatórios para o usuário final, gerando constantes solicitações de mudança de escopo. A implementação pode ser complexa se envolver múltiplos pesos e critérios.
    *   **Impacto:** Insatisfação do usuário, constante necessidade de refatoração da lógica de negócio, estouro de prazo/orçamento.

3.  **Qualidade e Volume dos Dados de Entrada (PDFs):**
    *   **O que é:** Falta de conhecimento sobre a qualidade (legibilidade, consistência) e o volume esperado dos PDFs que serão processados diariamente/mensalmente.
    *   **Por que é risco:** PDFs de baixa qualidade (escaneados, cortados, baixa resolução) dificultarão a extração automática. Um volume alto exigirá uma arquitetura escalável e resiliente, que pode não estar prevista.
    *   **Impacto:** Erros de extração, falha no processamento, gargalos de performance, necessidade de investimento em infraestrutura não orçado.

4.  **Ausência de Processo de Validação e Tolerância a Erro:**
    *   **O que é:** Não há clareza sobre como os dados extraídos serão validados e qual a tolerância do cliente para erros na extração.
    *   **Por que é risco:** Extrações automáticas, especialmente de PDFs despadronizados, raramente atingem 100% de acurácia. A falta de um "humano no loop" ou de um limite aceitável de erro pode levar a dados incorretos no sistema sem detecção.
    *   **Impacto:** Dados incorretos afetando a funcionalidade de filtragem, decisões erradas dos usuários, reputação do sistema comprometida.

5.  **Ampla Gama de Requisitos Não Investigados (RED Flags):**
    *   **O que é:** Pontos críticos como autenticação/autorização, integrações com sistemas externos, volume de usuários, requisitos de LGPD, prazos reais e planos de operação pós-entrega não foram abordados.
    *   **Por que é risco:** Cada uma dessas áreas representa um potencial "projeto bomba" em si e, se não mapeada, resultará em surpresas significativas durante o desenvolvimento e a operação.
    *   **Impacto:** Incerteza generalizada, estouro massivo de escopo, prazos irrealistas, custos não previstos, problemas legais (LGPD), e um sistema que não atende às necessidades operacionais do cliente.

## 4. PERGUNTAS AINDA SEM RESPOSTA

As seguintes áreas críticas ainda precisam de investigação profunda antes de qualquer comprometimento contratual:

*   **Clareza do Cliente / Escopo:** O cliente tem um protótipo, mockups ou um documento detalhado das regras de negócio para a filtragem? Quem são os usuários finais e qual o seu número?
*   **Integrações Externas:** Há necessidade de integrar com outros sistemas do cliente ou APIs de terceiros (ex: sistemas ERPs, APIs de operadoras de saúde para validação ou atualização)?
*   **Autenticação / Segurança:** Como os usuários farão login? O sistema precisa de múltiplos perfis de usuário com permissões diferentes? Há requisitos de SSO ou integração com outros sistemas de autenticação?
*   **Volume / Performance:** Qual o volume estimado de PDFs a serem processados diariamente/mensalmente? Quantos usuários simultâneos o sistema deve suportar? Qual a latência aceitável para o processo de upload/extração e para a busca/filtragem?
*   **Prazo / Capacidade:** Qual o prazo real que o cliente espera para a entrega da primeira versão? Esse prazo inclui testes, homologação e implantação ou apenas desenvolvimento? O cliente tem disponibilidade para validações rápidas?
*   **Operação Pós-entrega:** Quem será o responsável pela manutenção dos extratores de PDF no futuro (lidar com novos layouts, alterações)? Quem fará a validação dos dados extraídos em produção?
*   **IA / Taxa de Erro:** Se a extração usar IA/ML, qual a taxa de erro aceitável pelo cliente? Quais as consequências financeiras/legais de um erro na extração?
*   **IA / Humano no Loop:** Haverá um processo de revisão humana dos dados extraídos antes de serem considerados "oficiais"? Ou o processo é 100% automatizado?
*   **IA / Dados de Treinamento:** Se for usada IA/ML, o cliente possui dados históricos de PDFs já com os campos extraídos e validados para treinamento inicial e contínuo? Como a equipe terá acesso a esses dados?
*   **LGPD / Dados Pessoais:** O sistema armazenará dados pessoais sensíveis (ex: CPF, histórico médico, nome completo)? Quais são os requisitos de conformidade com a LGPD e o setor de saúde?
*   **LGPD / Auditoria:** Existe necessidade de logar e auditar todas as ações dos usuários e do sistema (ex: quem acessou quais dados, quando um PDF foi processado)?

A chance de estouro de prazo, orçamento e insatisfação do cliente é altíssima. É imperativo obter clareza nas "Perguntas Ainda Sem Resposta" para mitigar os riscos e precificar o projeto de forma realista. A complexidade da extração de PDFs e da lógica de filtragem sozinha já justifica uma fase de diagnóstico técnico mais aprofundada ou, no mínimo, um Discovery pago para detalhar estes pontos.

## 1. NÍVEL DE COMPLEXIDADE

**Classificação:** Alto. Potencial de 🚨 Bomba se as expectativas de acurácia e o prazo não forem alinhados com a realidade da extração de PDFs despadronizados.

**Justificativa:** O projeto envolve a extração de dados de PDFs, com a complicação crítica de existir "cerca de 50 layouts diferentes". Isso transforma a extração de uma tarefa simples em um desafio complexo de engenharia, exigindo abordagens robustas de OCR e processamento de linguagem natural (NLP) ou machine learning (ML), ou uma quantidade massiva de regras específicas para cada layout. Cada variação de layout representa um risco potencial de quebra da extração e demanda tempo significativo para desenvolvimento e validação. Além disso, a lógica de "filtrar os 3 que se encaixariam melhor" requer um entendimento aprofundado das regras de negócio e pode envolver algoritmos de recomendação complexos, dependendo da granularidade dos critérios de correspondência.

## 2. STACK E ARQUITETURA RECOMENDADAS

### 2.1 Tecnologias Específicas
*   **Processamento e Extração de PDFs (Back-end Worker):**
    *   **Linguagem:** `Python` devido ao vasto ecossistema de bibliotecas para processamento de texto, visão computacional (OCR), machine learning e NLP.
    *   **Bibliotecas de OCR e Análise de Layout:** `Tesseract` (para OCR base) e `OpenCV` (para pré-processamento de imagem e detecção de estrutura). Para extração de tabelas, `Camelot` ou `Tabula-py`. Para extração de texto bruto, `PyPDF2` ou `pdfminer.six`. Para layouts mais complexos ou semi-estruturados, `AWS Textract`, `Google Document AI` ou `Azure Form Recognizer` devem ser avaliados, pois são serviços gerenciados com maior acurácia e escalabilidade, potencialmente reduzindo o esforço de engenharia para os 50 layouts. **A escolha entre customizado e serviço de nuvem dependerá do orçamento, prazo e acurácia exigida.**
    *   **Framework para API:** `FastAPI` (Python) ou `Flask`: Leve, performático e excelente para construção de APIs RESTful. A tipagem de dados do FastAPI é valiosa para garantir a consistência das entradas e saídas.
    *   **Fila de Mensagens:** `RabbitMQ` ou `Redis` com `Celery` (Python): Essencial para desacoplar a submissão de PDFs da extração. As extrações de PDFs são processos longos e CPU-intensivos que devem ser executados assincronamente em workers separados para não bloquear a API principal.
*   **Banco de Dados:**
    *   `PostgreSQL`: Banco de dados relacional robusto e maduro, ideal para armazenar os dados estruturados dos planos de saúde extraídos, perfis de usuário e informações de controle da extração (status, erros). Sua capacidade de JSONB e busca textual (full-text search) pode ser útil.
*   **Front-end (Web):**
    *   `React` ou `Vue.js`: Frameworks modernos para construção de interfaces de usuário reativas e componentizadas. Necessário para a interface de upload de PDFs, visualização dos planos extraídos e o formulário/interface de filtragem.

### 2.2 Arquitetura de Componentes
1.  **Interface de Usuário (Front-end Web):** Permite o upload dos PDFs, visualização do status das extrações, e a interface para filtrar e visualizar os planos recomendados.
2.  **API Gateway / Back-end Principal (FastAPI/Flask):** Recebe requisições do front-end (upload, consulta de status, filtragem). Responsável pela autenticação, validação inicial de requisições e orquestração. Para uploads de PDFs, a API faria um pré-processamento leve (e.g., armazenamento temporário em um object storage como S3) e enviaria uma mensagem para a fila. Para filtragem, a API consultaria o banco de dados e aplicaria a lógica de negócio.
3.  **Fila de Mensagens (RabbitMQ/Redis):** Buffer entre a API e os workers de extração. Armazena as tarefas de extração de PDFs a serem processadas.
4.  **Workers de Extração de PDF (Celery Workers):** Processos separados que consomem mensagens da fila. Cada worker pega um PDF, aplica as regras de OCR/Análise de Layout/ML, extrai os dados, valida-os e os armazena no banco de dados. Este componente será o mais complexo e intensivo em recursos.
5.  **Banco de Dados (PostgreSQL):** Armazena: dados brutos dos PDFs (se necessário para auditoria), dados estruturados extraídos dos planos de saúde, perfis de usuário e suas preferências, e metadados das extrações (status, erros, etc.).
6.  **Object Storage (S3 compatível):** Para armazenamento dos PDFs originais e de artefatos de processamento (e.g., imagens pré-processadas).

**Fluxo Básico:** Usuário faz upload de PDF -> Front-end -> API -> Armazena PDF em S3 -> Envia mensagem para Fila de Mensagens -> Worker de Extração consome mensagem -> Worker baixa PDF do S3 -> Processa/Extrai dados -> Armazena dados extraídos no PostgreSQL -> Atualiza status da extração (no PostgreSQL). Usuário consulta planos -> Front-end -> API -> Consulta PostgreSQL -> Aplica lógica de filtragem -> Retorna planos.

### 2.3 Possíveis Armadilhas de Arquitetura
*   **Monolítico de Extração:** Tentar criar um único extrator "inteligente" que lide com os 50 layouts de forma genérica sem uma modularização adequada para cada grupo de layouts ou sem considerar a manutenção das regras quando um layout mudar. Isso levará a um código "espaguete" difícil de manter e escalar.
*   **Sincronismo na Extração:** Não usar fila e workers assíncronos. Fazer a extração de PDF diretamente na requisição HTTP da API resultará em timeouts, travamento da aplicação e péssima experiência do usuário.
*   **Ignorar Pré-processamento e Pós-processamento:** Subestimar a necessidade de pré-processar os PDFs (ajuste de imagem, remoção de ruído) antes do OCR e de pós-processar os dados extraídos (validação de formato, normalização, limpeza) antes de armazenar.
*   **Falta de Human-in-the-Loop:** Projetar a solução com a expectativa de 100% de acurácia automática. Com 50 layouts e documentos despadronizados, a ausência de um mecanismo de revisão e correção humana dos dados extraídos levará a dados incorretos no banco e desconfiança do sistema.

### 2.4 Alternativas Consideradas e Descartadas
*   **Extração 100% Manual ou Baseada em Regex Simples:** Descartada devido à alta quantidade de layouts (50). Regras regex se tornariam impraticáveis de manter e frágeis a pequenas alterações nos layouts. Serviços manuais seriam proibitivamente caros e lentos para o volume esperado.
*   **Soluções "Low-Code" de RPA para Extração:** Embora existam ferramentas de RPA que prometem extração de PDFs, para 50 layouts diferentes e a necessidade de integração profunda com um banco de dados e lógica de filtragem, uma solução customizada oferece maior flexibilidade, controle e escalabilidade a longo prazo. Ferramentas RPA poderiam ser consideradas para partes muito específicas, mas não como a base da solução.

## 3. PRINCIPAIS RISCOS E PONTOS DE ATENÇÃO

1.  **Acurácia da Extração de PDFs Despadronizados (50 layouts):**
    *   **Risco:** A existência de 50 layouts diferentes para os PDFs é o *maior risco técnico*. Cada layout pode exigir regras de extração específicas, o que aumenta drasticamente a complexidade de desenvolvimento, o tempo de implementação e os testes. A acurácia da extração será um desafio constante; é improvável atingir 100% de precisão de forma automática, especialmente se os PDFs tiverem variações de qualidade (scans ruins, fontes diferentes, tabelas desalinhadas).
    *   **Impacto:** Dados incorretos no banco de dados, falhas na filtragem de planos, necessidade de retrabalho manual, insatisfação do cliente, estouro de prazo e orçamento para desenvolvimento e manutenção.

2.  **Validação e Tolerância a Erro na Extração:**
    *   **Risco:** Não há informação sobre como os dados extraídos serão validados e qual a tolerância do cliente para erros. Quem será o "humano no loop" para revisar e corrigir dados extraídos incorretamente? Se a expectativa é de alta acurácia sem revisão humana, o projeto é inviável no prazo comercial.
    *   **Impacto:** Se os dados extraídos não forem validados ou forem imprecisos, a funcionalidade de "filtrar os 3 melhores planos" será comprometida, levando a recomendações erradas e perda de confiança no sistema.

3.  **Definição da Lógica de Filtragem e "Melhor Perfil":**
    *   **Risco:** A definição de "os 3 que se encaixariam melhor no perfil" é subjetiva e ainda não foi detalhada. Essa lógica pode ser simples (filtros básicos) ou extremamente complexa (algoritmos de pontuação, ponderação de critérios, ML) e pode ser difícil de extrair de forma clara do cliente.
    *   **Impacto:** Se as regras de negócio não forem claras e bem definidas, haverá re-trabalho contínuo na implementação, insatisfação do cliente com os resultados da filtragem, e estouro de escopo.

4.  **Manutenção e Evolução dos Layouts de PDF:**
    *   **Risco:** Planos de saúde são documentos dinâmicos. Novos layouts podem surgir, e layouts existentes podem ser alterados pelas operadoras sem aviso. Não está claro como a solução irá lidar com essas mudanças futuras.
    *   **Impacto:** A cada nova versão ou novo layout, será necessário um esforço de desenvolvimento e teste para ajustar o extrator, o que pode gerar custos de manutenção não previstos e "quebras" inesperadas na produção.

5.  **Acesso aos Dados Reais (50 layouts):**
    *   **Risco:** Não foi confirmado se o cliente tem acesso e pode fornecer *todos* os 50 layouts de PDF (e suas variações, incluindo exemplos de documentos com problemas de qualidade) para fins de desenvolvimento e teste. A falta de exemplos representativos resultará em um extrator frágil que quebrará em produção.
    *   **Impacto:** Descoberta de layouts inesperados ou variações em produção, atrasando a entrega, aumentando o custo e comprometendo a qualidade da extração.

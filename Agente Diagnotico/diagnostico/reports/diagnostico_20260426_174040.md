## 1. NÍVEL DE COMPLEXIDADE

**Classificação:** 🚨 Bomba

**Justificativa:** O projeto apresenta uma série de riscos críticos que, combinados, o tornam uma "bomba" em potencial. A expectativa irrealista de 100% de acurácia em extração de informações contratuais por IA, sem qualquer validação humana, é o principal fator de risco técnico e legal. Soma-se a isso a intenção de integrar o sistema com uma conta pessoal do Google Drive para dados sensíveis, violando princípios básicos de segurança e privacidade. A responsabilidade total pela infraestrutura de comunicação (WhatsApp API) e a falta de clareza ou ponto técnico do cliente em áreas críticas (segurança, LGPD, autenticação) elevam o nível de complexidade e risco para o CITi a patamares inaceitáveis sob as condições atuais.

## 2. STACK E ARQUITETURA RECOMENDADAS

### 2.1 Tecnologias Específicas

*   **Extração e Processamento de PDF:**
    *   **Python com PyMuPDF (fitz) ou pypdf:** Para extração robusta de texto de PDFs nativos. Justificativa: Ferramentas maduras e eficientes para lidar com PDFs textuais, essenciais para a fase inicial de ingestão de documentos.
    *   **Framework de RAG (Retrieval-Augmented Generation) - Ex: LlamaIndex ou LangChain:** Para indexação, segmentação (chunking), embedding e recuperação de informações dos documentos. Justificativa: Necessário para preparar o texto para o LLM, garantindo que as respostas sejam fundamentadas nos contratos.
    *   **Modelo de Embeddings - Ex: OpenAI Embeddings, Cohere Embeddings ou modelo open-source (ex: BGE-large-en):** Para converter os "chunks" de texto em vetores numéricos. Justificativa: Fundamental para a busca semântica no banco de dados vetorial.
    *   **Banco de Dados Vetorial - Ex: ChromaDB (local para escala pequena) ou Pinecone/Weaviate (nuvem para escala futura):** Para armazenar e consultar os embeddings de forma eficiente. Justificativa: Essencial para a fase de "retrieval" do RAG, buscando os trechos mais relevantes do contrato.
*   **Chatbot Core (Backend):**
    *   **Python com FastAPI:** Para construir a API do chatbot, gerenciar requisições do WhatsApp e orquestrar as interações com o RAG e o LLM. Justificativa: Framework moderno, assíncrono e de alta performance, ideal para APIs leves.
    *   **Serviço de LLM (Large Language Model) - Ex: OpenAI GPT-4, Anthropic Claude, ou modelo open-source como Llama 3 (via API ou self-hosted):** Para gerar as respostas em linguagem natural. Justificativa: A escolha dependerá do balanço entre custo, performance, segurança de dados e capacidade de atender à expectativa (irrealista) de 100% de acurácia. Para dados sensíveis, um modelo self-hosted ou API com rigorosos acordos de privacidade é preferível.
*   **Integração com WhatsApp:**
    *   **SDK/API do provedor de WhatsApp Business - Ex: Twilio, MessageBird ou Meta Business Platform API:** Para receber e enviar mensagens. Justificativa: Necessidade de integração direta com o WhatsApp, que exige um provedor autorizado.
*   **Gerenciamento de Documentos (Alternativa ao Google Drive pessoal):**
    *   **Storage de objetos (ex: AWS S3, Google Cloud Storage, ou MinIO self-hosted):** Para armazenar os arquivos PDF de forma segura. Justificativa: Oferece segurança, escalabilidade e controle de acesso muito superiores a uma conta pessoal do Google Drive.
    *   **Interface Web administrativa (ex: Flask-Admin ou uma aplicação simples em React/Vue com FastAPI backend):** Para upload e gerenciamento controlado dos PDFs. Justificativa: Garante que apenas usuários autorizados e processos controlados manipulem os contratos, essenciais para segurança e LGPD.
*   **Banco de Dados Relacional - Ex: PostgreSQL:** Para armazenar metadados dos contratos, histórico de conversas, logs e configurações do sistema. Justificativa: Banco de dados robusto e amplamente utilizado para dados estruturados.

### 2.2 Arquitetura de Componentes

1.  **Interface do Usuário (WhatsApp):** O proprietário interage com o chatbot através do WhatsApp. Mensagens de texto são enviadas e recebidas via API do WhatsApp Business.
2.  **Módulo de Gateway WhatsApp:** Responsável por receber webhooks do provedor de WhatsApp (e.g., Twilio) e encaminhar as mensagens para o `Chatbot Core`. Também envia as respostas do `Chatbot Core` de volta para o WhatsApp.
3.  **Chatbot Core (API FastAPI):** É o cérebro do sistema.
    *   Recebe a pergunta do usuário.
    *   Orquestra a busca de informações via RAG.
    *   Formata o prompt para o LLM com as informações relevantes.
    *   Recebe a resposta do LLM e a encaminha de volta para o `Módulo de Gateway WhatsApp`.
    *   Gerencia o estado da conversa e o histórico.
4.  **Módulo de Gerenciamento de Documentos (Interface Admin / Upload Seguro):**
    *   Uma interface web segura onde um funcionário autorizado (não o proprietário diretamente) pode fazer upload, remover ou substituir os PDFs de contrato.
    *   Após o upload/remoção, dispara um evento para o `Módulo de Processamento de Documentos`.
    *   Os PDFs são armazenados em um `Storage de Objetos` seguro (não Google Drive pessoal).
5.  **Módulo de Processamento de Documentos (Worker Assíncrono):**
    *   Monitora o `Storage de Objetos` ou é acionado por eventos do `Módulo de Gerenciamento de Documentos`.
    *   Extrai texto dos PDFs (usando `PyMuPDF`).
    *   Segmenta o texto em "chunks" e gera embeddings para cada "chunk" (usando `Modelo de Embeddings`).
    *   Armazena os embeddings no `Banco de Dados Vetorial`.
    *   Armazena metadados do contrato (nome, ID, status) no `Banco de Dados Relacional`.
6.  **Banco de Dados Vetorial:** Armazena os embeddings para busca semântica.
7.  **Serviço de LLM:** Modelo de Linguagem Grande que recebe prompts formatados e gera respostas.
8.  **Banco de Dados Relacional (PostgreSQL):** Armazena histórico de conversas, metadados dos contratos, logs e configurações do sistema.

**Fluxo de Dados:**
Usuário (WhatsApp) -> Gateway WhatsApp -> Chatbot Core -> RAG (Vector DB + LLM) -> Chatbot Core -> Gateway WhatsApp -> Usuário (WhatsApp)

Gerenciamento de Contratos: Admin -> Módulo Gerenciamento Docs -> Storage Objetos -> Módulo Processamento Docs -> Banco de Dados Vetorial + Relacional

### 2.3 Possíveis Armadilhas de Arquitetura

*   **Dependência excessiva da "facilidade" de extração:** A crença de que contratos são "fáceis" e sem subjetividade pode levar a uma arquitetura que não lida bem com variações sutis na linguagem ou com a necessidade de inferência. Mesmo que o cliente diga que não há, a realidade dos documentos jurídicos é diferente.
*   **Expor diretamente o LLM:** Permitir que o LLM gere respostas sem validação de regras de negócio ou de segurança (guardrails) é extremamente perigoso, especialmente com a exigência de 100% de acurácia e dados sensíveis.
*   **Acoplamento forte entre módulos:** Misturar a lógica de integração do WhatsApp com a lógica de negócio do chatbot ou a lógica de processamento de IA tornará o sistema rígido e difícil de manter/escalar.
*   **Subestimar a infraestrutura de LLM:** Escolher um LLM open-source para self-hosting pode subestimar os requisitos de hardware (GPUs) e o custo/complexidade de gerenciamento.
*   **Tratar o gerenciamento de documentos como um "detalhe":** A integração com Google Drive (pessoal) ou qualquer método não controlado para dados sensíveis é um risco gigantesco e não pode ser minimizado.

### 2.4 Alternativas Consideradas e Descartadas

*   **Processamento manual/pré-indexação humana:** Descartado, pois o cliente espera que a IA processe e responda diretamente.
*   **Chatbot baseado em regras (rule-based):** Descartado, pois a expectativa é de que a IA "entenda e responda perguntas", o que implica em capacidades de linguagem natural de um LLM, não apenas correspondência de padrões.
*   **Desenvolver uma solução escalável e multi-tenant desde o início:** Descartado explicitamente pelo cliente, que solicitou foco apenas nos 12 contratos atuais. No entanto, é importante registrar que essa decisão limita drasticamente a reutilização futura do código para o plano de "vender para imobiliárias".

## 3. PRINCIPAIS RISCOS E PONTOS DE ATENÇÃO

1.  **Acurácia de 100% e Ausência de Validação Humana (ALERTA CRÍTICO):**
    *   **O que é:** O cliente exige 100% de acurácia nas respostas da IA sobre informações contratuais, mas não prevê nenhum mecanismo de validação humana (nem na homologação, nem em produção).
    *   **Por que é risco:** A IA generativa não garante 100% de acurácia, especialmente com textos complexos como contratos, podendo "alucinar" ou interpretar erroneamente.
    *   **Impacto:** Risco altíssimo de fornecer informações incorretas com consequências legais, financeiras e de reputação para o cliente, com responsabilidade recaindo sobre nossa equipe.

2.  **Uso de Conta Pessoal do Google Drive com Dados Sensíveis (ALERTA CRÍTICO):**
    *   **O que é:** O plano é que o sistema acesse uma pasta do Google Drive associada à conta pessoal da proprietária, contendo contratos com informações sensíveis. Não há um método de autenticação/autorização definido.
    *   **Por que é risco:** Exposição de dados pessoais da proprietária e de terceiros para o nosso sistema, sem controle de segurança adequado. Dificuldade de gerenciar permissões e potencialmente acessar dados não relacionados ao projeto.
    *   **Impacto:** Violação da privacidade, risco de vazamento de dados, não conformidade com LGPD e outros regulamentos de proteção de dados.

3.  **LGPD e Conformidade com Dados Sensíveis (ALERTA CRÍTICO):**
    *   **O que é:** Os contratos contêm informações sensíveis. As práticas propostas (100% acurácia sem validação, GDrive pessoal, ausência de ponto técnico de segurança do cliente) são incompatíveis com as exigências da LGPD.
    *   **Por que é risco:** Falta de um plano claro de conformidade com a LGPD em todas as etapas do projeto.
    *   **Impacto:** Multas pesadas, ações judiciais, danos à reputação e paralisação do projeto para adequação legal.

4.  **Responsabilidade Total pela Aquisição e Configuração da API do WhatsApp (ALERTA CRÍTICO):**
    *   **O que é:** Nossa equipe será responsável por todo o processo de aquisição, configuração e gestão da API do WhatsApp Business, sem suporte técnico ou conta existente do cliente.
    *   **Por que é risco:** Dependência de terceiros (Meta, provedores), processos burocráticos e técnicos complexos, custos não mapeados.
    *   **Impacto:** Atrasos significativos no cronograma, custos adicionais para o projeto e nossa equipe assumindo responsabilidades de infraestrutura externa ao escopo usual de desenvolvimento.

5.  **Complexidade Oculta da Linguagem Contratual (RED FLAG):**
    *   **O que é:** Embora o cliente descreva os contratos como "normais" e sem cláusulas subjetivas, a natureza da linguagem jurídica frequentemente exige interpretação de contexto e inferência, que a IA pode ter dificuldade em realizar com 100% de acurácia.
    *   **Por que é risco:** A IA pode falhar ao responder perguntas que exigem mais do que uma extração literal, mesmo em textos "não rebuscados". A expectativa do cliente pode não corresponder à capacidade da IA.
    *   **Impacto:** Alta taxa de "erros de leitura" ou "alucinações" em cenários que o cliente não considera complexos, levando a insatisfação e necessidade de ajustes constantes na IA.

6.  **Gerenciamento Ambíguo de PDFs e Falta de Ponto Técnico do Cliente (RED FLAG):**
    *   **O que é:** A forma de gerenciamento dos PDFs via Google Drive pessoal não é clara em termos técnicos e não há um ponto de contato técnico do cliente para discutir requisitos de integração, segurança e autenticação.
    *   **Por que é risco:** Nossa equipe precisaria fazer suposições sobre a implementação, o que pode levar a soluções inseguras, inadequadas ou incompatíveis com a infraestrutura/políticas do cliente.
    *   **Impacto:** Retrabalho, vulnerabilidades de segurança e falhas na integração do módulo de gerenciamento de documentos.

7.  **Clareza e Maturidade do Cliente (RED FLAG):**
    *   **O que é:** O projeto possui um "prazo indefinido" ("poderia ser em 1 ano"), "custo a ser avaliado pela solução", e o cliente apresenta um baixo nível de entendimento técnico sobre os requisitos.
    *   **Por que é risco:** Um prazo tão elástico pode levar a falta de prioridade, inação, ou, paradoxalmente, a mudanças bruscas de escopo e expectativas infladas. A falta de conhecimento técnico dificulta a validação e a tomada de decisões.
    *   **Impacto:** Desgaste da equipe, dificuldade em gerenciar o escopo e o cronograma, e risco de o projeto nunca ser entregue ou ter suas prioridades alteradas drasticamente.

## 5. RECOMENDAÇÃO FINAL

**Decisão:** ❌ Não fechar sem antes resolver os pontos críticos de segurança, conformidade e expectativa de acurácia.

**Justificativa:** A combinação de uma exigência irrealista de "100% de acurácia" para informações contratuais com **dados sensíveis**, a total **ausência de validação humana**, a proposta de **integrar com uma conta pessoal do Google Drive** para esses dados e a nossa responsabilidade total pela **infraestrutura de WhatsApp** criam um cenário de risco extremo. Esses fatores tornam o projeto inviável e legalmente perigoso sob as condições atuais. É imperativo que esses pontos críticos sejam endereçados, compreendidos e resolvidos com o cliente antes de qualquer compromisso. O CITi não pode assumir riscos tão altos de responsabilidade legal e técnica.

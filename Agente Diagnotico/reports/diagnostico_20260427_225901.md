## 1. NÍVEL DE COMPLEXIDADE

**Classificação:** 🚨 Bomba

**Justificativa:** Este projeto, conforme diagnosticado, apresenta uma combinação de requisitos técnicos de altíssima complexidade e riscos de negócio inaceitáveis para o prazo e o nível de definição atual. A extração de dados de *fotos* de documentos (requisito imprescindível) utilizando IA, aliada à necessidade de preencher formulários bancários via APIs de terceiros (bancos), com um "risco legal" explícito em caso de erro, configura um cenário de engenharia complexa e delicada. A ausência de um fluxo de usuário claro, a expectativa de um prazo de 2 meses para *todas* as etapas (desenvolvimento, testes, homologação, implantação) e a expectativa inicial de "sem manutenção" após a entrega, indicam um desalinhamento crítico entre a percepção do cliente e a realidade técnica do projeto.

## 2. STACK E ARQUITETURA RECOMENDADAS

### 2.1 Tecnologias Específicas
*   **Camada de Ingestão e Pré-processamento de Imagens:** OpenCV para processamento de imagem (normalização, correção de perspectiva, binarização) e Pillow para manipulação de imagens Pythonicas. Essencial para otimizar a qualidade das fotos antes da IA.
*   **Camada de Extração de Dados (IA/OCR):** Python com Tesseract (OCR) combinado com bibliotecas como spaCy ou Transformers (para modelos NLP menores e específicos, se necessário para extração de entidades) para pós-processamento e validação semântica dos dados extraídos. Para maior acurácia e redução de complexidade de treinamento, serviços de OCR em nuvem como AWS Textract ou Google Cloud Vision AI *devem ser considerados*, mas requerem análise de custo e viabilidade. A escolha dependerá do trade-off entre custo, acurácia e a viabilidade de treinamento de modelos customizados para os documentos padronizados.
*   **Backend da Aplicação (Orquestração e CRM):** Python com FastAPI. A escolha se justifica pela necessidade de uma API leve e de alta performance para gerenciar o fluxo de documentos, orquestrar a extração de IA, armazenar dados do CRM e integrar com as APIs bancárias. O Pydantic integrado facilita a validação robusta de dados de entrada e saída, crucial para um sistema com risco legal.
*   **Banco de Dados:** PostgreSQL. É um banco de dados relacional robusto e escalável, ideal para armazenar dados de clientes (CRM), metadados de documentos, resultados da extração da IA e logs (caso sejam implementados).
*   **Armazenamento de Documentos:** S3-compatible object storage (ex: AWS S3, MinIO) para guardar as fotos originais e os PDFs. Essencial para escalabilidade, durabilidade e acesso seguro aos arquivos.
*   **Filas de Mensagens:** Redis Queues ou RabbitMQ. Necessário para desacoplar o processo de upload da foto da extração da IA e do preenchimento bancário, permitindo que essas operações de longa duração sejam processadas assincronamente e falhas possam ser reprocessadas.

### 2.2 Arquitetura de Componentes
1.  **Módulo de Ingestão de Documentos:** API REST que recebe as fotos dos documentos. Valida formato e envia para um serviço de fila.
2.  **Serviço de Processamento de Imagem:** Consome da fila, pré-processa as imagens (OpenCV) para melhorar a qualidade e envia para o módulo de IA.
3.  **Módulo de Extração por IA (OCR/NLP):** Consome imagens pré-processadas, executa o OCR, extrai as entidades necessárias e envia os resultados para a base de dados, marcando o status como "pendente de revisão".
4.  **Módulo de CRM:** Gerencia dados dos clientes, associa documentos e dados extraídos, e gerencia o estado dos processos.
5.  **Módulo de Validação Humana:** Interface web onde o "administrador" (ou revisor) visualiza a imagem original lado a lado com os dados extraídos pela IA, corrige eventuais erros e aprova a extração.
6.  **Módulo de Integração Bancária:** Recebe os dados validados, mapeia para os campos específicos de cada API bancária e realiza o preenchimento. Deve ter um mecanismo de re-tentativa e log de cada transação.
7.  **Serviço de Filas (para processamento assíncrono):** Para upload, extração, e preenchimento bancário, garantindo resiliência.
8.  **Interface de Usuário (Admin/Revisor):** Aplicação web para interação do administrador com o CRM, upload de documentos, validação e acompanhamento dos processos.

**Fluxo de Dados Primário:**
`Upload Imagem (UI)` -> `API Ingestão` -> `Fila (processamento imagem)` -> `Processamento Imagem` -> `Fila (IA)` -> `Módulo IA (OCR/NLP)` -> `DB (dados extraídos + status "pendente")` -> `UI Validação Humana` -> `DB (dados aprovados + status "aprovado")` -> `Módulo Integração Bancária` -> `APIs Bancárias`.

### 2.3 Possíveis Armadilhas de Arquitetura
*   **Subestimar a variabilidade das fotos:** Assumir que as fotos "padronizadas" sempre terão qualidade suficiente levará a baixa acurácia da IA e muito retrabalho manual. O pré-processamento de imagens é crítico.
*   **Fragilidade das integrações bancárias:** A dependência de APIs de terceiros (bancos) que podem mudar sem aviso ou ter baixa disponibilidade. Sem um tratamento robusto de erros, retentativas e monitoramento, o sistema será instável.
*   **Acoplamento rígido entre módulos:** Tentar fazer a extração de IA e o preenchimento bancário de forma síncrona com o upload da imagem, resultando em timeouts e falhas complexas de gerenciar. As filas são essenciais.
*   **Falta de observabilidade:** Sem logging detalhado, monitoramento e rastreabilidade de ações (auditoria), diagnosticar problemas será impossível, especialmente com risco legal.
*   **Construir o "CRM" inteiro:** Se a ideia de CRM for além do gerenciamento de documentos e status de preenchimento, o escopo pode explodir rapidamente.

### 2.4 Alternativas Consideradas e Descartadas
*   **Web Scraping/RPA para bancos:** Descartado devido à afirmação do cliente de que existem APIs oficiais, o que é uma abordagem mais estável e recomendada. Caso as APIs se mostrem inviáveis ou não existam para todos os bancos, essa seria uma alternativa a ser reavaliada, mas com altíssimo custo e risco de manutenção.
*   **Uso de um CRM de mercado (ex: Salesforce, Zoho CRM):** Não foi explorado em profundidade na entrevista, mas seria uma alternativa se o "CRM" do projeto for mais complexo do que um simples gerenciamento de processos. No entanto, a integração profunda de IA e automação bancária exigiria um alto grau de customização nesses CRMs, potencialmente elevando custos. Para o escopo aparente de "apenas admin", a construção de um módulo específico dentro da nossa aplicação para gerenciamento de clientes/processos é mais direta.
*   **IA 100% On-Premise (sem serviços de nuvem):** Descartado como primeira opção para OCR de fotos se a acurácia se tornar um problema. Embora a recomendação inicial inclua Tesseract/OpenCV (open source), para fotos, a acurácia de serviços de nuvem como AWS Textract é geralmente superior e pode reduzir o tempo de desenvolvimento inicial para o modelo de IA, compensando o custo por requisição.

## 3. PRINCIPAIS RISCOS E PONTOS DE ATENÇÃO

1.  **🚨 ALERTA CRÍTICO: Expectativa de Acúcia da IA vs. Risco Legal:** O cliente espera "menos erros possível" e enfrenta "risco legal" com dados incorretos. Extrair dados de *fotos* (mesmo que de documentos padronizados) é inerentemente propenso a erros. A IA não atingirá 100% de acurácia. A falta de um processo de validação humana *obrigatório e bem definido* é uma receita para problemas legais e retrabalho massivo.
2.  **🚨 ALERTA CRÍTICO: Prazo de 2 Meses:** Este prazo é completamente inviável para um projeto dessa complexidade. Ele não permite tempo adequado para design, desenvolvimento de IA (incluindo curadoria de dados para treinamento/teste), integrações bancárias, testes robustos (unitários, integração, end-to-end), homologação com o cliente e implantação. É o caminho mais rápido para um projeto "bomba" e estouro de prazo/orçamento.
3.  **🚨 ALERTA CRÍTICO: Ausência de Log de Auditoria:** Com "risco legal" envolvido e preenchimento de formulários bancários, a ausência de um log de auditoria para rastrear quem fez o quê (upload, revisão, envio) é um ponto de falha gravíssimo que expõe o cliente a sérias vulnerabilidades em caso de auditoria, disputa ou fraude.
4.  **⚠️ RED FLAG: Fotos de Documentos como Input:** A qualidade e variabilidade das fotos impactarão diretamente a acurácia da IA. "Alguns" exemplos de fotos são insuficientes para um treinamento e validação robustos para cenários reais de produção. Haverá uma alta demanda por dados de treinamento e um processo contínuo de avaliação e possível retreinamento.
5.  **⚠️ RED FLAG: Falta de Clareza no Fluxo de Usuário e UX:** A ausência de protótipos ou fluxos definidos significa que o cliente ainda está descobrindo "o que quer". Isso resultará em constantes mudanças de requisitos e designs durante o desenvolvimento, consumindo o prazo já irreal.
6.  **⚠️ RED FLAG: Expectativa de "Sem Manutenção":** Nenhum sistema de software, especialmente um com IA e integrações com terceiros, pode ser "entregue pronto" e funcionar sem manutenção. APIs bancárias mudam, o desempenho da IA pode degradar, bugs são descobertos e vulnerabilidades de segurança surgem. O cliente precisa entender o ciclo de vida do software e os custos associados a ele, mesmo que em contrato separado.
7.  **Dependência de Terceiros (APIs Bancárias):** Embora existam APIs, sua documentação pode não ser completa/atualizada, ambientes de sandbox podem ser limitados, e a estabilidade em produção pode variar. Isso é uma dependência fora do nosso controle direto.

## 4. PERGUNTAS AINDA SEM RESPOSTA

1.  **Validação Humana:** Qual é o processo detalhado de validação humana? Quem executa, quais campos são revisados, qual o tempo limite para revisão? O que acontece se a IA falha criticamente ou o humano não revisa a tempo?
2.  **Acurácia Aceitável:** Qual é a taxa de acurácia mínima *quantitativa* que o cliente aceita para a IA de extração de dados de fotos, considerando o "risco legal"?
3.  **Dados de Treinamento/Teste:** O cliente se compromete a fornecer um volume *significativo e diversificado* de fotos de documentos para treinamento e teste da IA, cobrindo variações de iluminação, ângulo e qualidade da imagem?
4.  **APIs Bancárias:** Quantos bancos exatamente? Qual o status da documentação para *cada uma* dessas APIs? O cliente possui credenciais e acesso a *ambientes de sandbox* para todos eles? Existe um contato técnico em *cada banco* para suporte?
5.  **Log de Auditoria:** Dada a discussão sobre "risco legal", o cliente realmente entende e aceita os riscos de não ter um log de auditoria detalhado? É crucial reavaliar esse requisito.
6.  **Escopo Exato do "CRM":** O "CRM" se limita ao gerenciamento de documentos e status de preenchimento, ou inclui outras funcionalidades como gestão de leads, funil de vendas, histórico de comunicação, etc.?
7.  **Orçamento Realista:** Qual é o novo intervalo de orçamento *realista* que o cliente está disposto a considerar após compreender a complexidade do projeto?

## 5. RECOMENDAÇÃO FINAL

**Decisão:** ❌ Não fechar sem antes resolver X

**Justificativa:** Este projeto, em seu estado atual, é uma "bomba" em potencial com altíssima probabilidade de estourar prazos, orçamentos e gerar insatisfação grave do cliente devido a expectativas irrealistas. As principais áreas de risco são a IA de extração de fotos, a integração com APIs bancárias, a ausência de logs de auditoria para um projeto com risco legal, a falta de definição de escopo e, criticamente, o prazo e orçamento completamente descolados da realidade.

Recomendo **não fechar o contrato** sem antes:
1.  **Reajustar o prazo e o orçamento** para valores que reflitam a real complexidade do projeto (mínimo de 4-6 meses para uma primeira versão viável com foco nos riscos críticos).
2.  **Definir e documentar claramente o processo de validação humana** da IA e a taxa de acurácia aceitável.
3.  **Implementar um sistema de Log de Auditoria** para todas as ações relevantes do sistema, devido ao risco legal.
4.  **Realizar uma fase de Discovery mais aprofundada** com o cliente para detalhar o fluxo de usuário, obter todos os exemplos de documentos necessários e validar a viabilidade das APIs bancárias. Isso pode ser oferecido como um projeto inicial separado.

Sem essas resoluções, o projeto representa um risco inaceitável para a reputação e saúde financeira da CITi.
